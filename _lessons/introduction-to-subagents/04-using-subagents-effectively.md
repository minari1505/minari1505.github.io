## 학습 목표

- 서브에이전트가 도움이 되는 때와 방해가 되는 때 구분하기
- 연구·코드 리뷰·커스텀 system prompt가 왜 좋은 사용처인지 이해하기
- 피해야 할 안티패턴(전문가 페르소나, 순차 파이프라인, test runner) 알기
- "중간 작업이 중요한가?"라는 판단 규칙 적용하기

## 서브에이전트가 빛나는 때

핵심은 하나 — **탐색이 실행과 분리**될 때입니다. 각 단계가 이전 단계의 발견에 의존한다면 메인 스레드에 두어야 합니다. 하지만 **답만 필요하고 과정은 상관없다면** 위임하세요.

서브에이전트가 뛰어난 작업:

- 과정의 중계가 아니라 **결과**가 필요할 때
- 탐색 작업이 메인 스레드의 context를 어지럽힐 때
- **새로운 시선**이나 커스텀 system prompt가 도움이 될 때

### 연구(Research) 작업

전형적 사용처입니다. 낯선 코드베이스의 인증 방식을 조사할 때, 메인 스레드는 JWT가 어디서 검증되는지 알면 되지 검색한 모든 파일을 볼 필요는 없습니다. research 서브에이전트가 수십 개 파일을 읽고 함수 호출을 추적한 뒤, 깔끔한 요약만 돌려줍니다:

```text
JWT 검증은 middleware/auth.js 42번째 줄에서 일어나며,
route/api.js의 Express 라우터에서 호출됩니다.
```

### 코드 리뷰

Claude는 코드가 **다른 사람이 쓴 것으로 제시될 때** 더 효과적으로 리뷰합니다. 메인 스레드에서 여러 턴에 걸쳐 만든 기능을 같은 스레드에 리뷰시키면, 만드는 데 관여했기에 새로운 눈으로 보기 어렵습니다. reviewer 서브에이전트는 별도 context에서 `git diff`를 실행하고 수정 파일을 읽어, 작성 이력 없이 리뷰 기준을 적용합니다. system prompt에 팀 고유 리뷰 기준을 담아 일관성도 확보할 수 있습니다.

### 커스텀 system prompt

Claude Code의 기본 prompt는 간결·코드 중심 응답을 지향합니다. 그것이 안 맞는 경우:

- **Copywriting 서브에이전트** — 톤·독자·스타일 지침을 줌. 랜딩 페이지나 이메일 캠페인엔 기본 기술적 문체가 부적합.
- **Styling 서브에이전트** — 디자인 시스템 파일을 가리키면, 실행 시 색상 변수·간격·컴포넌트 패턴이 자동으로 context에 로드됨.

## 서브에이전트가 해로운 때 (안티패턴)

서브에이전트를 띄우는 오버헤드(가시성 상실, 요약 압축)는 **메인 스레드가 못 하는 무언가**를 할 때만 정당화됩니다.

- **전문가 주장(Expert claims)** — "너는 Python 전문가야" 같은 prompt는 가치가 없습니다. Claude는 이미 그 지식이 있고, 전문가 서브에이전트가 메인 스레드보다 더 할 수 있는 건 없습니다.
- **순차 파이프라인(Sequential pipelines)** — 버그 재현→디버그→수정 같은 3-에이전트 흐름. 작업이 진정 독립적일 때만 되고, 각 단계가 이전 발견에 의존하면(버그 수정은 거의 항상 그렇습니다) 핸드오프에서 정보가 손실됩니다.
- **Test runner** — 테스트 실패 시 진단하려면 전체 출력이 필요한데, "tests failed"만 돌려주면 세부를 얻으려 추가 스크립트를 만들어야 합니다. 테스트 결과 이 패턴이 모든 구성 중 가장 나빴습니다.

## 판단 규칙

서브에이전트를 쓸지 정할 때 한 가지만 물으세요: **"중간 작업이 중요한가?"**

- **아니오** (최종 결과만 필요) → 서브에이전트에 위임
- **예** (과정을 보고 반응해야 함) → 메인 스레드에 유지

## 핵심 정리

- **쓰기 좋은 곳**: 연구·탐색, 코드 리뷰, 커스텀 system prompt가 필요한 작업.
- **피할 곳**: 실제 능력을 더하지 않는 "전문가" 페르소나, 단계가 서로 의존하는 다단계 파이프라인, 전체 출력이 필요한 테스트 실행.
- 판단 규칙은 "중간 작업이 중요한가?" — 아니면 위임, 맞으면 메인 스레드.
