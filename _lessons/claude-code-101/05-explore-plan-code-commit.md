## 학습 목표

- Explore → Plan → Code → Commit 4단계 workflow의 흐름 이해하기
- 각 단계에서 무엇을 하고 어디서 방향을 바로잡아야 하는지 파악하기
- 코딩 단계를 매끄럽게 만드는 성공 기준·도구·테스트의 역할 이해하기

## 이 강의의 핵심

이 과정에서 딱 하나만 가져간다면 바로 이 workflow입니다: **Explore, Plan, Code, Commit.** 이 흐름 없이 대부분은 곧장 Claude에게 코드를 작성하라고 시키는데, 그러면 나중에 더 많이 되돌려야 합니다.

## Explore와 Plan

앞 두 단계를 가장 빠르게 처리하는 방법은 **Plan Mode**입니다. Plan Mode에서 Claude는 파일을 수정할 수 없고, 구현을 어떻게 할지 정보를 모으기 위해 파일을 읽기만 합니다.

Shift + Tab을 눌러 텍스트 입력 아래 "Plan Mode"가 보이게 한 뒤 이렇게 씁니다.

```text
이미지 업로드 파이프라인에 WebP 변환을 추가해야 해. 파이프라인의 어느 지점에서 처리해야 하는지,
새 의존성이 필요한지, 어떻게 접근할지 파악해줘.
```

Claude가 관련 파일을 읽고 웹 검색을 하며 행동 계획을 제시합니다. 검토해서 기준에 맞는지 판단하고, 아니면 특정 부분을 수정해달라고 요청합니다. **코드가 작성되기 전**이라 방향을 바로잡기 가장 좋은 지점입니다. 변경할 의도 없이 코드베이스 요약만 원한다면 Plan Mode가 아니어도 explore 서브에이전트를 실행할 수 있습니다.

## Code

계획이 좋아 보이면 "approve"를 선택해 수락하고 Claude가 항목을 처리하게 합니다. 파일 수정을 자동 수락할지 매번 물어볼지 선택할 수 있습니다.

Claude는 계획을 "완료"로 보기 전에 최선을 다해 문제를 해결하지만, 때로는 사람이 개입해야 합니다. Plan Mode의 장점은, 실행 후에도 결과에 이른 맥락을 함께 갖고 있어 Claude의 다음 결정을 잘 이끌 수 있다는 점입니다.

코딩 단계를 매끄럽게 만드는 팁:

- **성공 기준을 정의하기.** Claude가 결과를 확신하려면 "무엇이 올바른가"가 분명해야 합니다. 계획을 쓸 때 이를 명시하세요.
- **도구를 추가하기.** 목표 달성을 돕는 도구는 왕복을 크게 줄입니다. 예를 들어 웹 UI를 만든다면 Claude in Chrome 확장을 설치해 Claude Code가 브라우저 탭을 제어하며 UI를 직접 테스트하게 할 수 있습니다.
- **테스트 스위트를 포함하기.** Claude가 계속 검증할 수 있는 테스트를 주세요. Claude가 테스트를 대신 작성할 수도 있습니다. 다만 넘기기 전에, 거짓 양성을 피하도록 테스트가 믿을 만한 기준인지 확인하세요.

> 팁: Claude가 같은 문제에 계속 부딪힌다면, 그 해결책을 CLAUDE.md 파일에 저장하라고 요청하세요.

## Commit

변경을 직접 테스트해 만족스러우면 코드를 push할 차례입니다. 커밋 전에 **서브에이전트 code reviewer**를 실행해 작업을 검토하게 하세요. 서브에이전트는 코드베이스를 새로운 시선으로 보며, 세션 내내 쌓인 main agent의 편향을 갖지 않습니다. 그런 다음 Claude에게 내 스타일에 맞는 커밋 메시지를 생성하게 하고, 이 과정을 반복합니다.

## 핵심 정리

Claude Code를 효과적으로 쓰려면 Explore → Plan → Code → Commit workflow를 따르세요.

- **Explore**는 프로젝트에 필요한 관련 맥락을 Claude에게 줍니다.
- **Plan**은 Claude가 성공을 측정할 기준이 되는 행동 계획을 만듭니다.
- **Code**는 최종 결과에 도달하기까지 사용자와 Claude가 주고받는 과정입니다.
- **Commit**은 코드를 검토·push해 다음 기능으로 넘어가게 합니다.
