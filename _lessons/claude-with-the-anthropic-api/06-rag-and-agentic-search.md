- RAG가 무엇이고 왜 필요한지 이해하기
- 청킹·임베딩·검색으로 이어지는 전체 RAG 흐름 파악하기
- 의미 검색(임베딩)과 어휘 검색(BM25), 멀티 인덱스 파이프라인 구분하기

## RAG란? (Retrieval Augmented Generation)

Claude는 학습 시점 이후의 정보나 **내 사설 데이터**(사내 문서, DB)를 모르는 상태예요. **RAG = 질문과 관련된 문서를 먼저 검색해서 프롬프트에 넣어준 뒤 답하게 하는 기법**이에요. 환각을 줄이고 근거 있는 답을 만들어요.

```
질문 → 관련 문서 검색 → [질문 + 검색된 문서]로 프롬프트 구성 → Claude가 근거 기반 답변
```

## 1. 텍스트 청킹 (Chunking)

긴 문서를 통째로 넣을 수 없으니 **작은 조각(chunk)**으로 나눠요.

- 너무 크면 관련 없는 내용까지 섞이고, 너무 작으면 맥락이 끊겨요.
- 문단·문장 경계로 자르고, 조각 간 **약간 겹치게(overlap)** 하면 경계에서 맥락이 유지돼요.

## 2. 텍스트 임베딩 (Embeddings)

각 청크를 **의미를 담은 벡터(숫자 배열)**로 변환해요. 의미가 비슷한 텍스트는 벡터 공간에서 가까이 위치해요. 질문도 같은 방식으로 벡터화한 뒤, **가장 가까운 청크**를 찾으면 그게 "의미적으로 관련된" 문서예요.

## 3. 전체 RAG 흐름

```
[준비 단계]  문서 → 청킹 → 임베딩 → 벡터 인덱스에 저장
[질의 단계]  질문 → 임베딩 → 인덱스에서 유사 청크 검색 → 프롬프트에 삽입 → Claude 답변
```

## 4. BM25 어휘 검색 (Lexical search)

임베딩(의미 검색)만으로는 **정확한 키워드·희귀 용어·코드·ID** 매칭에 약할 수 있어요. **BM25는 단어 빈도 기반의 고전적 어휘 검색**으로, 정확한 단어 일치에 강해요.

| 방식 | 강점 | 약점 |
|---|---|---|
| 임베딩(의미) | 의역·유사 개념 | 정확한 키워드·희귀어 |
| BM25(어휘) | 정확한 단어 일치 | 동의어·의역 |

## 5. 멀티 인덱스 RAG 파이프라인

두 방식을 **결합**하면 서로의 약점을 보완해요. 의미 검색 결과 + 어휘 검색 결과를 합치고 재정렬(re-rank)해서 최종 컨텍스트를 만들어요. 이런 하이브리드가 실전에서 가장 견고해요.

## 에이전틱 검색

RAG를 고정 파이프라인으로 쓰는 대신, **Claude에게 검색을 도구로 주고** 스스로 "무엇을, 몇 번 검색할지" 결정하게 할 수도 있어요(앞 레슨의 tool use / 웹 검색과 연결). 복잡한 질문을 여러 검색으로 쪼개 답할 수 있어요.

## 정리

- RAG = 관련 문서를 **검색해서 프롬프트에 넣고** 답하게 하기 → 근거 있는 답.
- 흐름: 청킹 → 임베딩 → 인덱스 저장 → 질의 시 유사 검색 → 삽입.
- **의미 검색(임베딩) + 어휘 검색(BM25)** 하이브리드가 가장 견고해요.
- Claude에게 검색을 도구로 맡기면 **에이전틱 검색**이 돼요.
