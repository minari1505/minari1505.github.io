## 워크플로우 vs 에이전트

워크플로우와 에이전트는 Claude 한 번의 요청으로 끝나지 않는 작업을 처리하기 위한 두 가지 전략입니다. 이 강의의 핵심 질문은 "누가 다음 단계를 결정하는가?"입니다.

| | 워크플로우(Workflow) | 에이전트(Agent) |
|---|---|---|
| 제어 흐름 | **내 코드**가 미리 정한 경로로 | **Claude**가 스스로 다음 행동 결정 |
| 예측 가능성 | 높음 (정해진 단계) | 낮음 (동적) |
| 적합한 작업 | 단계와 입력이 명확한 작업 | 미리 다 정의하기 어려운 개방형 작업 |
| 테스트 난이도 | 비교적 쉬움 | 더 어려움 |
| 사용자 경험 | 정해진 플로우에 강함 | 더 유연함 |

**워크플로우**는 특정 문제를 풀기 위해 Claude 호출과 일반 코드를 미리 정해진 순서로 연결하는 방식입니다. 사용자가 제공할 입력과 처리 흐름을 어느 정도 예측할 수 있을 때 적합합니다.

**에이전트**는 Claude에게 목표와 도구를 주고, Claude가 어떤 도구를 어떤 순서로 쓸지 직접 결정하게 하는 방식입니다. 사용자의 요청이 다양하고, 정확한 단계나 파라미터를 미리 알기 어려울 때 적합합니다.

실무에서는 먼저 단순한 구조부터 시작하는 것이 좋습니다.

```text
단일 호출 → 워크플로우 → 에이전트
```

대부분의 제품 문제는 단일 호출이나 워크플로우로 충분합니다. 사용자는 "에이전트를 썼는지"보다 "제품이 안정적으로 작동하는지"에 관심이 있습니다. 그래서 가능한 곳에서는 워크플로우를 먼저 고려하고, 정말 필요한 경우에만 에이전트로 복잡도를 올리는 것이 안전합니다.

## 워크플로우 패턴

워크플로우 패턴은 검증된 구현 레시피에 가깝습니다. 패턴 이름 자체가 문제를 해결해주지는 않지만, 비슷한 문제를 설계할 때 좋은 출발점이 됩니다.

### 1. 병렬화 (Parallelization)

병렬화는 하나의 복잡한 판단을 여러 독립적인 하위 판단으로 나누고, 각 하위 작업을 동시에 실행한 뒤 결과를 합치는 방식입니다.

예를 들어 사용자가 부품 이미지를 올리면 가장 적합한 재료를 추천하는 앱을 만든다고 해봅시다. 한 번의 프롬프트에서 금속, 폴리머, 세라믹, 복합재, 고무, 목재를 모두 비교하게 할 수도 있지만, Claude가 한 요청 안에서 너무 많은 기준을 동시에 고려해야 합니다.

병렬화 workflow에서는 같은 이미지를 여러 Claude 요청에 나눠 보냅니다.

- 금속 적합성 평가
- 폴리머 적합성 평가
- 세라믹 적합성 평가
- 복합재 적합성 평가
- 고무 적합성 평가
- 목재 적합성 평가

각 요청은 한 재료의 판단 기준에만 집중합니다. 마지막 aggregation 단계에서 각 분석 결과를 모아 최종 추천을 생성합니다.

```
이미지
  ├─ 금속 평가
  ├─ 폴리머 평가
  ├─ 세라믹 평가
  └─ ...
      ↓
결과 종합 → 최종 재료 추천
```

이 패턴의 장점은 다음과 같습니다.

- Claude가 한 번에 한 관점만 집중하므로 분석 품질이 좋아짐
- 각 하위 프롬프트를 독립적으로 테스트하고 개선할 수 있음
- 새 재료나 새 기준을 추가하기 쉬움
- 병렬 실행으로 전체 처리 시간을 줄일 수 있음

병렬화는 여러 기준, 여러 선택지, 여러 전문 영역을 독립적으로 검토해야 할 때 잘 맞습니다.

### 2. 체이닝 (Chaining)

체이닝은 큰 작업을 순서가 있는 작은 단계로 나누는 방식입니다. 앞 단계의 출력이 다음 단계의 입력이 됩니다.

```text
초안 생성 → 초안 검토 → 규칙 위반 수정 → 최종 다듬기
```

예를 들어 소셜 미디어 마케팅 도구가 자동으로 숏폼 영상을 만든다고 하면, 모든 일을 한 번의 프롬프트로 처리하기보다 다음처럼 나눌 수 있습니다.

1. 관련 트렌딩 주제 찾기
2. 가장 흥미로운 주제 선택
3. 주제 리서치
4. 짧은 영상 스크립트 작성
5. AI avatar와 text-to-speech로 영상 생성
6. 소셜 플랫폼에 게시

체이닝이 특히 유용한 상황은 긴 프롬프트에 많은 제약 조건이 들어갈 때입니다. 예를 들어 기술 글을 쓰게 하면서 동시에 "AI가 썼다고 말하지 말 것", "이모지 금지", "과하게 캐주얼한 표현 금지", "전문적인 톤 유지" 같은 조건을 모두 넣으면 Claude가 일부 조건을 놓칠 수 있습니다.

이럴 때는 두 단계로 나누는 편이 안정적입니다.

1. 먼저 글을 생성한다
2. 생성된 글을 다시 넣고, 규칙 위반만 집중적으로 수정하게 한다

```text
Revise the article below.
1. Remove any text that identifies the author as an AI.
2. Remove all emojis.
3. Replace casual or cringey writing with technical writing.
```

체이닝은 다음 상황에 잘 맞습니다.

- 복잡한 작업을 순서 있는 하위 작업으로 나눌 수 있음
- 각 단계 사이에 일반 코드 처리나 검증을 넣고 싶음
- 한 프롬프트에 모든 조건을 넣으면 Claude가 일부를 놓침
- 각 Claude 호출이 한 가지 일에 집중하길 원함

### 3. 라우팅 (Routing)

라우팅은 입력을 먼저 분류하고, 분류 결과에 맞는 전용 처리 경로로 보내는 방식입니다.

```
문의 접수 → 유형 분류 → (환불팀 / 기술지원 / 일반문의) 각 경로로
```

예를 들어 영상 스크립트 생성 도구에서 사용자가 `"programming"` 또는 `"surfing"` 같은 주제를 입력한다고 해봅시다. 프로그래밍 주제는 교육형 스크립트가 잘 맞고, 서핑 주제는 에너지 있는 엔터테인먼트형 스크립트가 더 적합할 수 있습니다. 한 프롬프트로 모든 주제를 처리하면 결과가 애매해지기 쉽습니다.

라우팅 workflow는 보통 두 단계입니다.

1. Claude가 입력을 predefined category 중 하나로 분류
2. 분류 결과에 맞는 전용 prompt template 또는 pipeline 실행

```text
Categorize the topic of a video into one of the listed categories:

<topic>Python functions</topic>

<categories>
- Educational
- Entertainment
- Comedy
- Personal vlog
- Reviews
- Storytelling
</categories>
```

Claude가 `"Educational"`이라고 답하면, 이후에는 교육형 스크립트 작성 프롬프트를 사용합니다.

라우팅은 다음 상황에 적합합니다.

- 요청 유형마다 다른 프롬프트, 도구, 모델, 정책이 필요함
- 카테고리를 명확히 정의할 수 있음
- Claude가 분류를 충분히 안정적으로 수행함
- 라우팅 비용보다 전문화된 처리의 이점이 더 큼

간단한 질문은 저렴하고 빠른 모델로, 어려운 질문은 더 강한 모델로 보내는 비용 최적화에도 사용할 수 있습니다.

### 4. Evaluator-Optimizer

Evaluator-Optimizer는 생성자와 평가자를 분리하고, 평가 결과를 다시 생성자에게 보내 개선을 반복하는 패턴입니다.

```text
입력 → Producer가 결과 생성 → Grader가 평가
                         ↑       ↓
                    피드백 반영 ← 불합격
                         ↓
                       합격 시 종료
```

강의 예시는 이미지에서 CAD 모델을 만드는 workflow입니다.

1. 사용자가 금속 부품 이미지를 업로드
2. Claude가 이미지를 보고 부품을 설명
3. Claude가 CadQuery 코드로 3D 모델 생성
4. 렌더링 이미지 생성
5. Claude가 원본 이미지와 렌더링을 비교 평가
6. 문제가 있으면 피드백을 반영해 다시 모델 생성

이 패턴은 "정답을 한 번에 만들기 어렵지만, 결과가 좋은지 평가할 수 있는 작업"에 강합니다. 코드 생성 후 테스트 실행, 문서 초안 작성 후 style guide 검사, 디자인 생성 후 요구사항 비교 같은 작업에도 적용할 수 있습니다.

## 에이전트 — 도구와 환경

에이전트는 Claude에게 목표와 도구 목록을 주고, Claude가 스스로 도구를 골라 목표에 도달하게 하는 구조입니다.

앞에서 만든 reminder 예시를 보면 에이전트의 성격이 잘 드러납니다.

- `get_current_datetime`: 현재 날짜와 시간을 가져옴
- `add_duration_to_datetime`: 특정 날짜에 기간을 더함
- `set_reminder`: 특정 시점에 알림 생성

각 도구는 단순하지만 Claude는 이들을 조합할 수 있습니다.

- `"What's the time?"` → `get_current_datetime`
- `"What day is it in 11 days?"` → `get_current_datetime` + `add_duration_to_datetime`
- `"Remind me to go to the gym next Wednesday"` → 현재 날짜 확인 + 날짜 계산 + reminder 생성

또 `"When does my 90-day warranty expire?"`처럼 필요한 정보가 빠진 요청에서는 구매일을 먼저 물어볼 수도 있습니다. 에이전트의 장점은 이런 동적 판단입니다.

### 도구는 너무 구체적이지 않게

좋은 에이전트 도구는 조합 가능해야 합니다. Claude Code가 좋은 예입니다. Claude Code에는 보통 다음처럼 범용적인 도구가 있습니다.

- `bash`: 명령 실행
- `read`: 파일 읽기
- `write`: 파일 생성
- `edit`: 파일 수정
- `glob`: 파일 찾기
- `grep`: 파일 내용 검색

반대로 `refactor_code`, `install_dependencies`, `fix_bug`처럼 과하게 특화된 도구를 많이 제공하지 않습니다. Claude는 범용 도구를 조합해 refactor, dependency 설치, 테스트 실행, 버그 수정 같은 복잡한 작업을 수행합니다.

에이전트용 도구를 설계할 때는 다음을 생각해야 합니다.

- Claude가 여러 방식으로 조합할 수 있는가?
- 실행 결과를 Claude가 확인할 수 있는가?
- 실패했을 때 에러 메시지가 충분히 구체적인가?
- 위험한 동작은 승인이나 제한을 거치게 되어 있는가?

예를 들어 소셜 미디어 영상 에이전트라면 다음 도구들이 조합 가능한 기본 단위가 될 수 있습니다.

- `bash`: FFmpeg로 영상 처리
- `generate_image`: 이미지 생성
- `text_to_speech`: 대사를 오디오로 변환
- `post_media`: 완성된 결과를 플랫폼에 업로드

이렇게 도구를 구성하면 단순한 "영상 생성 후 게시" workflow뿐 아니라, 샘플 이미지를 먼저 만들고 사용자 승인을 받은 뒤 전체 영상을 만드는 인터랙티브 흐름도 가능합니다.

## 환경 점검 (Environment Inspection)

에이전트가 제대로 동작하려면 자신의 행동 결과를 관찰할 수 있어야 합니다. Claude는 기본적으로 환경을 직접 볼 수 없으므로, 도구와 시스템 프롬프트가 "어떻게 확인할지"를 제공해야 합니다.

Computer Use가 대표적인 예입니다.

```text
화면 스크린샷 → Claude가 다음 클릭/입력 결정 → 실행 → 다시 스크린샷 → 다음 판단
```

Claude가 버튼을 클릭했을 때 페이지가 이동했는지, 메뉴가 열렸는지, 에러가 났는지는 다시 화면을 보기 전까지 알 수 없습니다. 그래서 행동 뒤 관찰이 필수입니다.

파일 작업도 마찬가지입니다. Claude가 파일을 수정하기 전에는 먼저 현재 내용을 읽어야 합니다.

```text
파일 읽기 → 구조 파악 → 수정 → 테스트/검색으로 결과 확인
```

코드를 읽지 않고 바로 수정하면 기존 구조나 import, 스타일을 깨뜨릴 가능성이 큽니다. 그래서 agent system prompt나 도구 설계에서 "쓰기 전에 읽기"를 자연스럽게 유도해야 합니다.

복잡한 생성 작업에서는 점검 도구를 명시적으로 제공할 수 있습니다. 예를 들어 영상 생성 에이전트라면:

- `whisper.cpp`로 자막과 timestamp를 생성해 대사 위치 확인
- FFmpeg로 일정 간격의 screenshot을 추출해 시각 요소 확인
- 생성 결과를 원래 요구사항과 비교

환경 점검을 설계할 때 던질 질문은 하나입니다.

> Claude가 이 행동이 성공했는지 어떻게 알 수 있는가?

실전 체크리스트는 다음과 같습니다.

- 파일 수정 전 현재 파일을 읽을 수 있는가?
- UI 조작 후 screenshot이나 DOM 상태를 볼 수 있는가?
- API 호출 후 상태 코드와 응답 body를 확인할 수 있는가?
- 생성물에 대해 테스트, lint, parser, evaluator를 실행할 수 있는가?
- 실패 시 다시 시도하거나 사용자에게 질문할 수 있는가?

환경 점검은 에이전트를 단순 실행기가 아니라, 결과를 보고 적응하는 시스템으로 만듭니다.

## 에이전트를 써야 할까? 판단 기준

에이전트는 유연하지만 비용과 불확실성이 큽니다. 워크플로우보다 성공률이 낮을 수 있고, 실행 경로가 매번 달라져 테스트와 관찰이 어렵습니다. 그래서 에이전트를 쓰기 전에 다음 기준을 확인해야 합니다.

| 기준 | 질문 |
|---|---|
| 복잡성 | 작업이 다단계이고, 미리 정확한 흐름을 정의하기 어려운가? |
| 가치 | 추가 비용과 지연을 감수할 만큼 결과 가치가 큰가? |
| 실현성 | Claude가 이 유형의 판단과 도구 사용을 잘 수행하는가? |
| 오류 비용 | 실패를 감지, 복구, 롤백할 수 있는가? |

하나라도 약하다면 단일 호출이나 워크플로우로 남는 편이 낫습니다. 특히 업무 흐름이 명확하고 UX가 정해져 있다면 워크플로우가 더 안정적입니다.

정리하면:

- 흐름을 알고 있다 → 워크플로우
- 흐름을 모른다 → 에이전트 후보
- 결과를 안정적으로 테스트해야 한다 → 워크플로우 우선
- 다양한 사용자 요청을 하나의 유연한 시스템으로 처리해야 한다 → 에이전트 고려

## 정리

- **워크플로우 = 내 코드가 제어**, **에이전트 = Claude가 제어**.
- 워크플로우는 예측 가능하고 테스트하기 쉬우며, 에이전트는 더 유연하지만 불확실성과 비용이 큽니다.
- 주요 workflow 패턴은 **병렬화**, **체이닝**, **라우팅**, **evaluator-optimizer**입니다.
- 에이전트 도구는 너무 특화하기보다 Claude가 조합할 수 있는 범용 단위로 설계하는 것이 좋습니다.
- 에이전트는 행동 후 환경을 관찰해야 합니다. 파일은 읽고 수정하고, UI는 조작 후 다시 보고, 생성물은 테스트나 evaluator로 확인해야 합니다.
- 가능한 곳에서는 워크플로우를 쓰고, 예측하기 어려운 개방형 작업에서만 에이전트를 고려합니다.
