## 학습 목표

- 워크플로우와 에이전트의 차이를 구분하기
- 대표 워크플로우 패턴(병렬·체이닝·라우팅)을 이해하기
- 에이전트가 도구·환경과 상호작용하는 방식과, 언제 에이전트를 쓸지 판단하기

## 워크플로우 vs 에이전트

가장 중요한 구분이에요! 🔑

| | 워크플로우(Workflow) | 에이전트(Agent) |
|---|---|---|
| 제어 흐름 | **내 코드**가 미리 정한 경로로 | **Claude**가 스스로 다음 행동 결정 |
| 예측 가능성 | 높음 (정해진 단계) | 낮음 (동적) |
| 적합한 작업 | 단계가 명확한 작업 | 미리 다 정의하기 어려운 개방형 작업 |

> **먼저 단순한 것부터.** 단일 호출 → 워크플로우 → 에이전트 순으로, 필요한 만큼만 복잡도를 올려요. 대부분은 단일 호출이나 워크플로우로 충분해요.

## 워크플로우 패턴

### 1. 체이닝 (Chaining)

작업을 **여러 단계로 쪼개** 순서대로 연결해요. 앞 단계의 출력이 다음 단계의 입력이 돼요. 각 단계가 단순해져 정확도가 올라가요.

```
초안 생성 → 초안 교정 → 최종 다듬기
```

### 2. 병렬화 (Parallelization)

독립적인 하위 작업을 **동시에** 실행해요. 속도가 빨라지고, 여러 관점의 결과를 모을 수 있어요.

- 예: 한 문서를 여러 기준으로 동시에 분석한 뒤 결과를 합침.

### 3. 라우팅 (Routing)

입력을 먼저 **분류**한 뒤, 종류에 맞는 처리 경로(또는 모델)로 보내요.

```
문의 접수 → 유형 분류 → (환불팀 / 기술지원 / 일반문의) 각 경로로
```

- 간단한 질문은 Haiku로, 어려운 건 Opus로 라우팅해 비용을 최적화할 수도 있어요.

## 에이전트 — 도구와 환경

에이전트는 **목표를 주면 Claude가 스스로 도구를 골라 쓰며** 목표에 도달해요. 앞 레슨의 도구 사용 + 에이전틱 루프가 기반이에요.

- **환경 점검(environment inspection)**: 에이전트는 행동 후 환경 상태(파일 변경, 명령 출력, 화면)를 관찰하고 다음 행동을 정해요. "행동 → 관찰 → 판단"을 반복해요.
- 도구를 잘 설계하는 게 에이전트 품질을 좌우해요. 되돌리기 어렵거나 승인이 필요한 동작은 전용 도구로 만들어 통제해요.

## 에이전트를 써야 할까? 판단 기준

네 가지를 모두 만족할 때 에이전트가 적합해요.

- **복잡성**: 다단계이고 미리 완전히 정의하기 어렵다
- **가치**: 높은 비용·지연을 감수할 만한 결과다
- **실현성**: Claude가 이 유형의 작업을 잘한다
- **오류 비용**: 실패를 감지·복구할 수 있다 (테스트·리뷰·롤백)

하나라도 "아니오"면 더 단순한 단계(단일 호출/워크플로우)로 남으세요.

## 정리

- **워크플로우 = 내 코드가 제어**, **에이전트 = Claude가 제어**.
- 워크플로우 패턴: **체이닝**(순차), **병렬화**(동시), **라우팅**(분류 후 분기).
- 에이전트는 "행동 → 환경 관찰 → 판단"을 반복하며, 도구 설계가 품질을 좌우.
- 단순한 것부터, 에이전트는 4가지 기준을 만족할 때만.
